# BS-001: Подтверждение отмены брони

## Пользовательская цель
Подтвердить намерение отменить активную бронь, избежав случайного нажатия на деструктивную кнопку.

## Контекст и триггер открытия
- **Родительский экран:** SCR-006 «Детали брони».
- **Триггер:** тап по CTA «Отменить» в активной брони со статусом `Pending` / `Paid`, при условии `now + 2h ≤ slot.startsAt`.
- **Тип шторки:** деструктивное подтверждение (destructive confirmation).

## Контент и данные на шторке
- Заголовок: «Отменить бронь?»
- Краткое описание: дата, время заезда, конфигурация трассы.
- Предупреждение: «Бронь будет отменена безвозвратно. Возврат депозита не предусмотрен в приложении (см. NFR-9).»
- Два действия:
    - **Основное (деструктивное):** «Отменить бронь» — отправляет запрос на бэкенд.
    - **Вторичное:** «Оставить бронь» — закрывает шторку без действий.
- Состояние загрузки во время отправки запроса (disabled-состояние кнопок).
- Inline-ошибка при отказе бэкенда (например, если окно отмены уже закрыто на стороне сервера).

## Связанные требования

### User Stories
- **US-8** — Как клиент, я хочу отменить бронь не позднее чем за 2 часа до старта, чтобы освободить место и не потерять деньги.

### Use Cases
- **UC-5** — Отмена брони клиентом: приложение проверяет условие `now + 2h ≤ startsAt` → открывает шторку подтверждения → при подтверждении отправляет запрос.

### Functional Requirements
- **FR-10** — Система должна позволять клиенту отменить бронь; кнопка «Отменить» активна только при `now + 2h ≤ slot.startsAt`.

### Non-Functional Requirements
- **NFR-1** — Бэкенд является источником истины; приложение корректно обрабатывает отказы (например, если окно отмены уже закрыто).
- **NFR-9** — Автоматические компенсации и возвраты не реализуются — это явно сообщается клиенту в тексте шторки.
