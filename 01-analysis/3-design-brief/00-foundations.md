# Фундамент дизайна приложения «Картинг-центр „Апекс“»

Общий каркас дизайн-решений, единый для всех экранов клиентского приложения.
Опирается на артефакты `01-analysis/2-requirements/` и фиксирует решения,
которые не должны переосмысляться на уровне отдельных экранов.

Источники: `01-business-requirements.md`, `02-functional-requirements.md`,
`03-non-functional-requirements.md`, `04-use-cases.md`, `05-user-stories.md`,
`../1-elicitation/domain-description.md`.

| Параметр | Значение | Обоснование |
|---|---|---|
| Тип продукта | Клиентское мобильное приложение | BR-2, NFR-10 |
| Целевые ОС | iOS и Android | NFR-3 (APNs/FCM — обе платформы) |
| Технологии | Кросс-платформенный фреймворк (Flutter / React Native) либо нативные клиенты — на этапе `3-design` уточняется совместно с командой разработки | Единая кодовая база ускоряет поставку к началу сезона (срок — «пару месяцев») |

## 2. Ключевые принципы UI

Принципы обязательны для всех экранов и служат критерием при приёмке макетов.

### 2.1. Mobile-first, «большой палец»
Все ключевые действия (выбор слота, бронь, отмена, оценка) — в нижней трети экрана,
доступны одной рукой. Верхняя часть — контекст и навигация.
*(US-2, US-5, US-8)*

### 2.2. Минимализм и фокус на одном действии
На экране — одна главная задача. Никаких «всё на одной странице»:
расписание отдельно, профиль отдельно, история отдельно.
Следствие ограничений скоупа: нет групповых броней, нет waivers, нет публичных
рейтингов, нет онлайн-эквайринга *(NFR-5, NFR-6, NFR-7, NFR-8, FR-18)*.

### 2.3. Доверие к бэкенду, прозрачность состояний
Приложение не валидирует «опытность» клиента, не считает скидки само,
не гарантирует наличие мест — оно честно показывает то, что отдаёт black-box
бэкенд, и корректно обрабатывает отказы *(NFR-1, NFR-2, NFR-11, FR-7, FR-11)*.

### 2.4. Ясность статуса
Любая сущность, имеющая жизненный цикл (бронь, слот), всегда показывает
свой текущий статус визуально: цветом, бейджем, текстом. Клиент никогда
не должен гадать, что происходит с его бронью *(FR-9, FR-17)*.

### 2.5. Предсказуемость цены
Итоговая цена показывается сразу, с раскрытием: базовая цена + применённая
скидка лояльности. Никаких «цена уточнится позже» *(FR-6, FR-7, US-7)*.

### 2.6. Доступность (a11y)
- Контрастность текста — не ниже WCAG AA (4.5:1).
- Все интерактивные элементы — не менее 44×44 pt/dp.
- Поддержка VoiceOver / TalkBack для ключевых сценариев (бронь, отмена, оценка).

## 3. Паттерны навигации

### 3.1. Глобальная навигация — нижний таб-бар
Три постоянных раздела (фиксированный набор, не расширяется в текущей поставке):

| Вкладка | Иконка | Назначение | Связанные UC |
|---|---|---|---|
| Расписание | calendar | Просмотр слотов, бронирование | UC-2, UC-3, UC-11 |
| Мои заезды | ticket | История и активные брони | UC-5, UC-9, UC-13 |
| Профиль | user | Телефон, лояльность, выход | UC-8 |

Авторизация (UC-1) — отдельный flow, не входит в таб-бар; после входа
пользователь попадает на вкладку «Расписание».

### 3.2. Локальная навигация
- **Внутри вкладки «Расписание»**: горизонтальный свайп по дням (7 дней по умолчанию) +
  фильтр дат для расширения горизонта *(FR-2, US-2)*.
- **Внутри вкладки «Мои заезды»**: сегмент-контрол с тремя состояниями
  (Активные / Состоявшиеся / Отменённые) *(FR-16, US-12)*.
- **Детали слота / брони**: модальный sheet (bottom sheet) снизу, не переход
  на отдельный экран — чтобы сохранить контекст расписания.
- **Оценка маршала**: модальное окно по push-уведомлению *(UC-7, FR-14)*.

### 3.3. Возвраты и жесты
- Системный «назад» (swipe от левого края / hardware back) работает везде,
  кроме flow авторизации и оплаты депозита.
- Закрытие модалок — свайпом вниз или крестиком в углу.

## 4. Универсальные состояния экранов

Каждый экран, работающий с данными бэкенда, обязан реализовывать все четыре
состояния. Это единый контракт для дизайнера и разработчика.

### 4.1. Загрузка (Loading)
- Первый вход на экран — **скелетон** (shimmer) по форме будущего контента.
- Обновление (pull-to-refresh, смена фильтра) — **inline-спиннер** без замены
  контента, чтобы не «мигать».
- Таймаут загрузки — 10 секунд, после чего переход в состояние «Ошибка».

### 4.2. Ошибка (Error)
- Иллюстрация + заголовок + короткое описание + **одна** главная кнопка
  («Повторить» или «Попробовать снова»).
- Текст — без технических подробностей. Примеры:
    - *«Нет соединения. Проверьте интернет и попробуйте ещё раз.»*
    - *«Не удалось загрузить расписание. Повторите через минуту.»*
    - *«Мест на этот заезд больше нет. Выберите другое время.»* *(FR-11, UC-11)*
- Критические ошибки (отклонённая оплата, невозможность отмены) —
  в виде bottom-sheet с понятной причиной и следующим шагом.

### 4.3. Пустота (Empty state)
Обязательны для всех списковых экранов. Всегда содержат:
1. Иллюстрацию / иконку.
2. Заголовок-ответ на вопрос «чего здесь нет?».
3. Подсказку, что делать дальше (call-to-action).

| Экран / состояние | Заголовок | Подсказка |
|---|---|---|
| Расписание, нет слотов на 7 дней | «Пока нет доступных заездов» | «Загляните позже — расписание обновляется» *(R-027)* |
| Слот, мест нет | «Мест на этот заезд нет» | «Выберите другое время» *(FR-11, UC-11)* |
| Мои заезды — Активные, пусто | «У вас пока нет активных броней» | «Посмотреть расписание» *(кнопка-ссылка)* |
| Мои заезды — Состоявшиеся, пусто | «История пуста» | «Ваш первый заезд появится здесь после оплаты» |
| Мои заезды — Отменённые, пусто | «Отменённых заездов нет» | — (без CTA) |

Лист ожидания не предлагается — это явное ограничение скоупа *(NFR-6, FR-11)*.

### 4.4. Успех (Success)
- Кратковременный toast / snackbar в верхней части экрана (2–3 секунды).
- Для значимых действий (бронь создана, бронь отменена, оценка отправлена) —
  inline-подтверждение внутри экрана с возможностью перейти к деталям.
- Примеры сообщений:
    - *«Бронь создана. Перейдите по ссылке, чтобы внести депозит.»* *(UC-3, UC-4)*
    - *«Бронь отменена. Место освобождено.»* *(UC-5)*
    - *«Спасибо за оценку!»* *(UC-7)*

## 5. Тон коммуникации (Tone of Voice)

Приложение общается с клиентом как **дружелюбный маршал на трассе**:
уверенно, коротко, по делу, без канцеляризмов и без панибратства.

| Аспект | Правило | Пример «да» | Пример «нет» |
|---|---|---|---|
| Длина фраз | Одна мысль — одно предложение | «Мест на этот заезд нет» | «К сожалению, на выбранный вами временной слот в настоящий момент свободные места отсутствуют» |
| От первого лица | Избегаем «мы», говорим о системе фактами | «Бронь отменена» | «Мы отменили вашу бронь» |
| Ошибки | Без вины пользователя, с решением | «Не получилось загрузить. Попробуйте ещё раз» | «Ошибка 500. Обратитесь в поддержку» |
| Отмена центром | Честно, с причиной, без драмы | «Заезд отменён из-за грозы. Причина: сильная гроза в районе трассы» *(FR-17)* | «Заезд не состоится по независящим от нас обстоятельствам» |
| Оценка маршала | Тёплый запрос, без давления | «Как вам заезд? Оцените маршала» | «Пожалуйста, обязательно оцените работу нашего сотрудника!» |
| Лояльность | Факт, а не маркетинг | «Ваш статус: Бронза. Скидка 10% применена» | «Поздравляем! Вы стали частью нашей дружной семьи!» |

Запрещённые приёмы:
- Манипулятивные формулировки («остался последний шанс», «только сегодня»).
- Избыточные восклицательные знаки (не более одного на экран).
- Технический жаргон в пользовательских сообщениях (никаких «депозит не верифицирован», «slot cancelled»).

## 6. Единый визуальный язык (базовые токены)

Конкретные цвета и шрифты фиксируются в дизайн-системе на следующем шаге
(`3-design/01-design-system.md`). Здесь — только принципы выбора:

- **Акцентный цвет** — ассоциация с картингом (динамика, скорость); используется
  только для CTA и активных состояний.
- **Семантические цвета** — статусы брони:
    - `Pending` — нейтральный/ожидание,
    - `Paid` — успех/подтверждение,
    - `Expired` / `CancelledByCenter` — приглушённый красный/серый.
- **Типографика** — один шрифтовой гарнитур, 4–5 размеров, чёткая иерархия.
- **Скругления** — умеренные, без «игривости» (продукт про сервис, не про игру).

## 8. Что НЕ входит в фундамент (и почему)

Следующие решения принимаются на уровне отдельных экранов или вынесены
за скоуп — чтобы не перегружать общий документ:

- Конкретные макеты экранов — в `3-design/02-screens/`.
- Дизайн-система (цвета, отступы, компоненты) — в `3-design/01-design-system.md`.
- Flow авторизации по SMS — специфичен для UC-1, описывается отдельно.
- Flow оплаты по `depositLink` — внешний переход, описывается в контракте API.
- Интерфейсы маршала и админа — вне скоупа текущей поставки (NFR-10).
