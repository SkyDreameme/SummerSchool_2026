# 5-mobile-app-spec — Спецификация клиентского приложения «Апекс»

Папка содержит финальные спецификации клиентского мобильного приложения картинг-центра «Апекс»: экраны, шторки, сквозные логики и сводный feature-list. Все артефакты построены на основе:

- `00-foundations.md` — фундамент дизайна (принципы UI, навигация, универсальные состояния, Tone of Voice)
- `design-brief.md` — реестр экранов и карта навигации
- `design-review.md` — ревью и разрешение противоречий
- OpenAPI-контракт (`api/`) — источник истины по эндпоинтам и моделям
- Требования `01–05` (FR, NFR, UC, US)

> **Статус:** Approved · Дата последней актуализации: 2026-07-05

---

## 🚀 Быстрый старт для разработчика

1. **С чего начать.** Прочитайте `00-foundations.md` — он задаёт единые правила для всех экранов (состояния Loading / Error / Empty / Success, Tone of Voice, a11y).
2. **Что реализовывать.** Откройте `feature-list.md` — там все 89 фич с приоритетами MoSCoW и ссылками на экраны/требования. MVP = 48 фич с приоритетом **Must**.
3. **Как устроена навигация.** См. `design-brief.md §2` — карта переходов между экранами.
4. **Детали конкретного экрана.** Откройте нужный `SCR-XXX.md` или `BS-XXX.md` — структура у всех одинаковая (§1–§13).
5. **Сквозной сценарий.** Если задача затрагивает несколько экранов (например, «создание брони»), откройте соответствующий `FLOW-XXX.md` из папки `09_Логики/`.
6. **API-контракт.** Все эндпоинты, модели и коды ответов — в `api/`. В каждом SCR/BS есть раздел §5.2 со ссылками на конкретные модули `api/<домен>/api.yaml`.
7. **Edge cases.** Перед сдачей экрана проверьте раздел §12 — там все сценарии из `design-review.md §3`, релевантные этому экрану.

## 🛠 Как пользоваться спецификацией

### Структура каждого SCR / BS

Все экраны и шторки следуют единой структуре (§1–§13 шаблона `_SCREEN_TEMPLATE.md`):

| Раздел | Что внутри | На что обратить внимание |
| --- | --- | --- |
| §1–§2 | Метаданные, история | Статус, дата обновления, связанные артефакты |
| §3 | Назначение и контекст | Пользовательская цель, бизнес-ценность, контекст открытия |
| §4 | Навигация | Входящие / исходящие переходы, системные жесты |
| §5 | Данные | Локальные модели, API-эндпоинты, кэширование |
| §6 | Структура экрана | Блоки, компоненты, CTA с условиями активации |
| §7 | Состояния | Loading / Error / Empty / Success + специфичные |
| §8 | Логика | Инициализация, happy path, бизнес-правила, валидация |
| §9 | Макет | Wireframe, адаптив, ориентация |
| §10 | Traceability | Ссылки на FR / NFR / US / UC |
| §11 | DoD | Чек-лист приёмки |
| §12 | Edge cases | Сценарии из `design-review.md §3` |
| §13 | Приложения | Смежные артефакты |

### Работа с API

- Все эндпоинты описаны в модулях `api/<домен>/api.yaml` (`auth/`, `slots/`, `bookings/`, `profile/`, `instructors/`).
- Модели — в `api/<домен>/models.yaml` и `api/common/models.yaml`.
- Корневой файл — `api/openapi.yaml` (ссылки на модули через `$ref`).
- В каждом SCR/BS (§5.2) и FLOW (§7) указаны конкретные эндпоинты и обрабатываемые коды ответов.
- **Бэкенд — источник истины (NFR-1).** Любое изменение состояния данных происходит на бэкенде; приложение только инициирует и отображает.

### Обновление артефактов

1. При изменении экрана — обновите `§2 История изменений` и `§1 Метаданные (дата последнего обновления)`.
2. При добавлении нового edge case — проверьте, не описан ли он уже в `design-review.md §3`.
3. При изменении API — сначала обновите контракт в `api/`, затем синхронизируйте SCR/BS/FLOW.
4. При добавлении новой фичи — обновите `feature-list.md` с указанием приоритета MoSCoW и traceability.

## ⚠️ Явные ограничения скоупа (Out of Scope)

Согласно `design-brief.md §3` и `design-review.md §5`, в текущей поставке **не реализуются**:

- Визуальная карта маршрута (NFR-8)
- Онлайн-эквайринг внутри приложения (NFR-5)
- Групповые брони и лист ожидания (NFR-6)
- Waivers и цифровые согласия (NFR-7)
- Валидация «опытности» клиента (NFR-8)
- Автоматические компенсации и возвраты (NFR-9)
- Интерфейсы ролей «Маршал» и «Администратор» (NFR-10)
- Публичный рейтинг маршала (FR-18)
- Текстовые комментарии к оценке (FR-14)
- Action-кнопки в Push-уведомлениях (FR-13)
- Смена номера телефона / удаление аккаунта
- Частичная / двойная оплата депозита

## 📝 Лицензия и соглашения

- Все артефакты — внутренняя документация проекта «Апекс».
- Шаблоны (`_SCREEN_TEMPLATE.md`, `_LOGIC_TEMPLATE.md`) обязательны для заполнения — не оставляйте плейсхолдеры `<…>` в финальных версиях.
- При расхождении между спецификацией экрана и API-контрактом — **источник истины API-контракт**. Сначала обновите контракт, затем SCR/BS/FLOW.
- При расхождении между спецификацией и `design-review.md` — сначала разрешите противоречие через обновление требований, затем синхронизируйте артефакты.
