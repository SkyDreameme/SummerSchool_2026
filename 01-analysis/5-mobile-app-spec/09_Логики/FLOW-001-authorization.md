3. Назначение и контекст
   3.1. Краткое описание
   Сквозной сценарий входа в приложение по номеру телефона через SMS-код. Используется при первом запуске, истечении JWT, принудительном логауте.

3.2. Пользовательская цель
Как клиент, я хочу войти по SMS, чтобы не запоминать пароли.

3.3. Бизнес-ценность
Минимальный порог входа → максимальная конверсия в первый просмотр расписания (BR-2).

3.4. Контекст запуска
Точка входа: старт приложения / 401 от бэкенда / тап «Выйти» в SCR-007
Сессия: не авторизован
Сеть: обязательна

5.3. Точки выхода

6. Граф переходов
   6.1. Happy path
1. Пользователь открывает приложение → SCR-001 (шаг 1).
2. Вводит номер телефона (+7, E.164).
3. Тапает «Получить SMS-код».
4. Приложение → POST /auth/sms/send → 202.
5. UI переходит на шаг 2, запускается таймер 60 с.
6. Пользователь вводит 4–6-значный код.
7. Приложение → POST /auth/sms/verify → 200 + accessToken.
8. Сохранить JWT → GET /clients/me → replace на TabBar.

6.2. Альтернативные пути
A1. SMS не приходит (3+ попытки)
Условие входа: attemptsLeft ≤ 0
Шаги:
1. Показать CTA «Не приходит код? Свяжитесь с поддержкой».
2. Пользователь тапает → open Telegram / tel:.
   Возврат в happy path: нет, флоу завершён.

A2. Повторная регистрация существующего номера
Условие входа: номер уже зарегистрирован
Шаги:
1. Бэкенд отправляет SMS, вход выполняется (FR-1, п. 14 из review).
   Возврат в happy path: с шага 5.

A3. Истёк JWT при повторном запуске
Условие входа: JWT в secure storage, но 401 на любой запрос
Шаги:
1. Очистить JWT → редирект на SCR-001.
   Возврат в happy path: с шага 1.

7. Используемые API
   | # | Метод | Endpoint | Назначение | Успешный код | Обрабатываемые ошибки |
   | --- | --- | --- | --- | --- | --- |
   | 1 | POST | api/auth/api.yaml → /auth/sms/send | Отправка SMS-кода | 202 | 400, 429 |
   | 2 | POST | api/auth/api.yaml → /auth/sms/verify | Проверка кода, получение JWT | 200 | 401, 429 |
   | 3 | GET | api/profile/api.yaml → /clients/me | Загрузка профиля | 200 | 401 |

7.1. Последовательность вызовов
Вызов #1 → при успехе → вызов #2 → при успехе → вызов #3.

7.2. Идемпотентность и повторные попытки
- Вызов #1: не идемпотентен (генерирует новый код). Повтор ограничен таймером 60 с + retryAfterSeconds из 429.
- Вызов #2: не идемпотентен (расходует попытки). Повтор при сетевой ошибке — да, с сохранением введённого кода.
- Вызов #3: идемпотентен (чтение). Повтор при ошибке — да.

8. Состояния данных
   | Сущность | Начальное состояние | Конечное состояние (успех) | Кто меняет |
   | --- | --- | --- | --- |
   | Client (JWT) | отсутствует | accessToken в secure storage | Бэкенд (выдаёт), клиент (хранит) |
   | Client (профиль) | — | phone, loyaltyStatus, discountPercent | Бэкенд |

8.1. Источники истины
JWT и профиль — на бэкенде. Клиент только хранит JWT в secure storage.

9. Обработка ошибок
   9.1. Сетевые ошибки
   | Ситуация | Поведение |
   | --- | --- |
   | Нет соединения при старте | Error state в SCR-001 + CTA «Повторить» |
   | Потеря связи на шаге 2 | Inline-ошибка «Нет соединения» + CTA «Повторить» |
   | Таймаут (> 10 с) | Error state |

9.2. Ошибки бэкенда
| Код | Смысл | Поведение |
| --- | --- | --- |
| 400 | Некорректный номер | Inline-ошибка «Проверьте номер телефона» |
| 401 | Неверный код | Inline-ошибка «Неверный код. Осталось попыток — N» |
| 429 | Rate limit | Таймер = retryAfterSeconds, CTA disabled |

9.3. Race conditions
| # | Сценарий | Обработка |
| --- | --- | --- |
| 1 | Двойной тап по «Получить SMS-код» | CTA disabled во время запроса |
| 2 | Двойной тап по «Подтвердить» | CTA disabled во время запроса |

10. Edge cases
    | # | Сценарий | Поведение |
    | --- | --- | --- |
    | 1 | Номер уже зарегистрирован (п. 14 из review) | Отправляем SMS, входим (не создаём нового клиента) |
    | 2 | SMS не доходит (3+ попытки, п. 15 из review) | CTA «Связаться с поддержкой» → Telegram / tel: |
    | 3 | Номер в международном формате (п. 16 из review) | Ограничиваем маской +7 |
    | 4 | Удалил приложение и установил заново (п. 17 из review) | JWT отсутствует → SCR-001 |
    | 5 | Смена номера / удаление аккаунта (п. 18 из review) | Out of Scope (NFR-7) |
    | 6 | 429 с retryAfterSeconds | Таймер = retryAfterSeconds |

10.1. Явные ограничения скоупа
- Смена номера телефона — вне скоупа (NFR-7).
- Удаление аккаунта — вне скоупа (NFR-7).
- Waivers и загрузка документов — вне скоупа (NFR-7).
