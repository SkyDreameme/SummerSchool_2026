3. Назначение и контекст
   3.1. Краткое описание
   Раздел с тремя вкладками: Активные / Состоявшиеся / Отменённые. Точка входа в детали брони и историю заездов.

3.2. Пользовательская цель
Клиент хочет видеть историю заездов и активные брони, чтобы отслеживать свои поездки.

3.3. Бизнес-ценность
Снижение нагрузки на администратора: клиент сам видит статус брони.

3.4. Контекст открытия
Сессия: авторизован
Сеть: обязательна
Предшествующее действие: тап по вкладке «Мои заезды» / возврат после бронирования / отмены

5.2. API-эндпоинты
| Метод | Endpoint | Назначение | Обрабатываемые коды |
| --- | --- | --- | --- |
| GET | api/bookings/api.yaml → /bookings?group=active | Активные брони | 200 |

| GET | api/bookings/api.yaml → /bookings?group=cancelled | Отменённые | 200 |

5.3. Кэширование и актуальность
TTL кэша: не кэшируется.
Стратегия: pull-to-refresh + push-инвалидация (при смене статуса брони).

6. Структура экрана
   6.1. Блок: Сегмент-контрол
   Назначение: переключение между вкладками
   Компоненты: `SegmentedControl` (Активные / Состоявшиеся / Отменённые)
   Данные: ActiveTab

6.2. Блок: Список броней
Назначение: карточки броней
Компоненты:
- `BookingCard` — дата, время, конфигурация, статус (цветовой индикатор)
- Статусы: Pending (нейтральный), Paid (успех), Expired (приглушённый), Completed (серый), CancelledByClient / CancelledByCenter (красный)
  Данные: BookingDetails[]

8. Логика и взаимодействия
   8.1. Инициализация
1. Загрузить GET /bookings?group=active.
2. Показать скелетон.
3. При успехе — отрисовать список.

8.2. Happy path
1. Пользователь открывает вкладку.
2. Приложение загружает список.
3. Пользователь тапает по карточке → SCR-006.

8.3. Бизнес-правила
| # | Правило | Обоснование |
| --- | --- | --- |
| 1 | Три вкладки соответствуют group=active/past/cancelled | FR-16 |

| 3 | Неизвестный статус — fallback на «Отменённые» + логирование | П. 27 из review |

12. Edge cases и ограничения
    | # | Сценарий | Поведение |
    | --- | --- | --- |
    | 1 | Бэкенд вернул неизвестный статус (Refunded) | Fallback на «Отменённые» + логирование (п. 27 из review) |
    | 2 | Сеть упала при pull-to-refresh | Оставить предыдущие данные, toast «Не удалось обновить» |
    | 3 | Бронь перешла в Completed автоматически | При следующем открытии — во вкладке «Состоявшиеся» (п. 4 из review) |

12.1. Явные ограничения скоупа
