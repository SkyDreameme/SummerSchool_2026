3. Назначение и контекст
   3.1. Краткое описание
   Экран входа в приложение по номеру телефона через SMS-код. Первый экран, который видит неавторизованный пользователь. Пароли не используются.

3.2. Пользовательская цель
Клиент хочет быстро войти в приложение по номеру телефона, чтобы не запоминать пароли.

3.3. Бизнес-ценность
Минимальный порог входа → максимальная конверсия в первый просмотр расписания (BR-2).

3.4. Контекст открытия
Сессия: не авторизован
Сеть: обязательна
Предшествующее действие: первый запуск приложения / истечение срока жизни JWT / принудительный логаут

6. Структура экрана
   Экран имеет два шага (step 1 — телефон, step 2 — код).

6.1. Блок: Шаг 1 — Ввод номера
Назначение: ввод номера телефона
Компоненты:
- `PhoneInput` — поле с маской +7 (XXX) XXX-XX-XX, автофокус
- `ButtonPrimary` «Получить SMS-код»
- `TextLink` «Пользовательское соглашение» (опционально, external)
  Данные: phone (локально)
  Видимость: всегда на шаге 1

6.2. Блок: Шаг 2 — Ввод кода
Назначение: ввод 4–6-значного SMS-кода
Компоненты:
- `CodeInput` — 4–6 ячеек, автоподтверждение при заполнении
- `Timer` «Отправить код повторно через N с»
- `ButtonSecondary` «Отправить код повторно» (активен после таймера)
- `TextLink` «Не приходит код? Свяжитесь с поддержкой» (появляется после 3 неудачных попыток — edge case #15 из review)
  Данные: code (локально), timerEndsAt, attemptsLeft
  Видимость: после успешного /auth/sms/send

6.3. CTA
| CTA | Тип | Активен при | Действие |
| --- | --- | --- | --- |
| «Получить SMS-код» | primary | phone валиден (E.164) | POST /auth/sms/send |
| «Отправить повторно» | secondary | timerEndsAt ≤ now | POST /auth/sms/send |
| «Связаться с поддержкой» | link | attemptsLeft ≤ 0 | open Telegram / tel: |

8. Логика и взаимодействия
   8.1. Инициализация
1. Проверить наличие JWT в secure storage.
2. Если JWT есть и не истёк → replace на TabBar.
3. Если JWT нет / истёк → показать шаг 1.

8.2. Happy path
1. Пользователь вводит номер телефона.
2. Тапает «Получить SMS-код».
3. Приложение → POST /auth/sms/send → 202.
4. UI переходит на шаг 2, запускается таймер 60 с.
5. Пользователь вводит код из SMS.
6. Приложение → POST /auth/sms/verify → 200 + accessToken.
7. Сохранить JWT → GET /clients/me → переход на TabBar.

8.3. Бизнес-правила
| # | Правило | Обоснование |
| --- | --- | --- |
| 1 | Повторная регистрация существующего номера = вход (не создаётся новый клиент) | FR-1, п. 14 из review |
| 2 | Таймер повторной отправки — 60 с | FR-1, п. 1 из review |
| 3 | Максимум 5 попыток ввода кода, затем блокировка + CTA «Поддержка» | п. 15 из review |
| 4 | Поддерживается только код +7 (Россия) | п. 16 из review |

8.4. Валидация
| Поле | Правило | Сообщение |
| --- | --- | --- |
| phone | E.164, начинается с +7, 11 цифр | «Проверьте номер телефона» |
| code | 4–6 цифр, pattern `^\d{4,6}$` | «Код должен содержать 4–6 цифр» |

12. Edge cases и ограничения
    | # | Сценарий | Поведение |
    | --- | --- | --- |
    | 1 | Номер уже зарегистрирован | Отправляем SMS, входим (FR-1, п. 14 из review) |
    | 2 | SMS не доходит (3+ попытки) | Показываем CTA «Связаться с поддержкой» (п. 15 из review) |
    | 3 | Номер в международном формате (не +7) | Ограничиваем маской +7 (п. 16 из review) |
    | 4 | Удалил приложение и установил заново | JWT отсутствует → показываем SCR-001 (п. 17 из review) |
    | 5 | Смена номера / удаление аккаунта | Out of Scope (п. 18 из review) |
    | 6 | 429 с retryAfterSeconds | Таймер = retryAfterSeconds, CTA disabled |

12.1. Явные ограничения скоупа
- Смена номера телефона — вне скоупа (NFR-7).
- Удаление аккаунта — вне скоупа (NFR-7).
- Waivers и загрузка документов — вне скоупа (NFR-7).
