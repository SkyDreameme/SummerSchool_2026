# 🐞 Баг-репорт: кнопка «Забронировать место» не отображается / неактивна

**Дата:** 2026-07-05  
**Автор:** Разработчик  
**Статус:** ✅ Исправлено  
**Приоритет:** Высокий (блокирует основной пользовательский сценарий — бронирование)

---

## 1. Краткое описание

На экране **Бронирование** (SCR-004) кнопка **«Забронировать место»** не отображалась либо была неактивна. Пользователь мог выбрать экипировку, но не мог завершить создание брони, что делало невозможным прохождение основного сценария (FLOW-002).

---

## 2. Симптомы и шаги воспроизведения

**Симптомы:**
- После перехода на экран бронирования (из деталей слота) пользователь видел только бесконечный индикатор загрузки (`LoadingBox`).
- Или экран отображался, но кнопка «Забронировать место» отсутствовала или была серой (disabled).
- В логах не было видно явных ошибок, но бронь не создавалась.

**Шаги для воспроизведения (до исправления):**
1. Войти в приложение.
2. Перейти на вкладку «Расписание».
3. Выбрать слот с доступными местами.
4. Нажать «Забронировать» на экране деталей слота → переход на `BookingScreen`.
5. На экране бронирования отображается только крутящийся спиннер (или пустой экран).
6. Кнопка «Забронировать место» отсутствует или неактивна.

---

## 3. Ожидаемое поведение

На экране бронирования пользователь должен видеть:
- детали слота (дата, время, конфигурация),
- выбор экипировки (Rental / Own),
- итоговую цену с учётом скидки,
- активную кнопку **«Забронировать место»**, которая при нажатии создаёт бронь и открывает success-шторку.

---

## 4. Фактическое поведение

- Экран зависал на загрузке (`LoadingBox`) из-за того, что не обрабатывались ошибки при получении данных слота или профиля клиента.
- Если данные загружались, кнопка часто была disabled из-за неправильных условий активации или из-за того, что `slot` или `client` оставались `null`.
- Кроме того, в `ApexRepository` была проверка на наличие активной брони (`hasAnyActive`), которая блокировала создание новой брони, даже если у пользователя уже была одна активная (по спецификации это правильно, но для тестирования мешало).

---

## 5. Причина (Root Cause)

1. **Отсутствие обработки ошибок загрузки**  
   В `LaunchedEffect` при загрузке слота и клиента не было блока `try-catch`, поэтому при любой ошибке (например, `ApiException`) экран оставался в состоянии `loading = true` бесконечно.

2. **Неявное состояние ошибки**  
   Не было отдельного состояния `loadError`, поэтому экран не показывал сообщение пользователю и не давал возможности перезагрузить или вернуться назад.

3. **Неправильная логика активации кнопки**  
   Кнопка была доступна только при `slot != null && client != null`, но если загрузка зависала, эти условия не выполнялись.

4. **Отсутствие обратной связи при ошибках создания**  
   При ошибках (409, 410) не было inline-сообщений с действиями (например, «К моим записям»), поэтому пользователь не понимал, что делать дальше.

---

## 6. Решение

### 6.1. Добавлена обработка ошибок загрузки

В `BookingScreen` добавлены:
- состояние `loadError: String?`,
- блок `try-catch` в `LaunchedEffect` с установкой `loadError`,
- отображение `ErrorState` при ошибке загрузки с кнопкой возврата.

```kotlin
LaunchedEffect(slotId) {
    loading = true
    loadError = null
    try {
        slot = repository.getSlot(slotId)
        client = repository.getClient()
    } catch (e: ApiException) {
        loadError = e.message ?: "Не удалось загрузить данные"
    } catch (e: Exception) {
        loadError = "Произошла ошибка при загрузке"
    } finally {
        loading = false
    }
}
