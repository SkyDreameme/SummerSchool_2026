# Крупные задачи — симптом, требования, реализация

Сводный документ по трём крупнейшим задачам (баг-фиксам) проекта «Апекс». Для каждой задачи: краткий симптом/цель, найденные требования (если есть в `01-analysis/2-requirements/`) и подтверждение, что задача реализована.

Источники: `bug-reports/01bag-report.md`, `bug-reports/02bag-report.md`, `bug-reports/03bag-report.md`.

## 1. Кнопка «Выйти» не отображается в профиле

**Симптом:** на экране «Профиль» (SCR-007) кнопка «Выйти» не отображалась после прокрутки экрана до конца — выйти из аккаунта было невозможно (только переустановка приложения или очистка данных вручную).

**Требования:** явного FR под выход из аккаунта нет — в `02-functional-requirements.md` эта функция отдельно не выделена (сессия/выход упоминается только косвенно, в связке с FR-1 — вход по SMS). Формальное требование не найдено.

**Реализовано:** ✅ да.
- Убран `PullToRefreshBox`, ломавший высоту `Column`; добавлен `Modifier.fillMaxSize()` на корневой контейнер.
- Кнопка «Выйти» теперь всегда прижата к низу экрана (`Spacer(Modifier.weight(1f))`).
- Добавлена резервная иконка выхода в `TopAppBar`, доступная даже при ошибке загрузки профиля.

## 2. Кнопка «Забронировать место» не отображается / неактивна

**Симптом:** на экране бронирования (SCR-004) пользователь либо видел бесконечный спиннер загрузки, либо кнопка «Забронировать место» была серой (disabled) — основной сценарий бронирования (FLOW-002) был заблокирован.

**Требования:** найдены прямые требования на создание брони:
- **FR-4** — выбор экипировки (Rental / Own) при бронировании.
- **FR-5** — ровно одна активная бронь на клиента (проверка `hasAnyActive`).
- **FR-8** — генерация `depositLink` после создания брони.
- **FR-9** — статусы брони (`Pending` и жизненный цикл).

**Реализовано:** ✅ да.
- Добавлены состояние `loadError` и `try-catch` в `LaunchedEffect`, чтобы экран не зависал в `loading = true` при ошибке загрузки слота/клиента.
- Добавлен `ErrorState` с возможностью вернуться назад.
- Исправлена логика активации кнопки (не зависит от «подвисшей» загрузки).
- Добавлены inline-сообщения с действиями для ошибок `409 NO_SEATS` / `410 SLOT_CANCELLED`.

## 3. Общий таймер отправки SMS для всех аккаунтов

**Симптом:** таймер повторной отправки SMS-кода (60 сек) был общим для всех номеров телефона — после отправки кода на один номер система блокировала повторную отправку и для другого номера, таймер не сбрасывался при выходе из аккаунта.

**Требования:** прямо связано с **FR-1** — вход клиента исключительно по номеру телефона через SMS-код. FR-1 задаёт саму механику SMS-входа, но не детализирует изоляцию таймера по номерам — это уточнение обнаружено только на уровне бага и тест-кейсов (`test-case.md`, TC-AUTH-001, TC-AUTH-002).

**Реализовано:** ✅ да.
- Единственное поле `smsSentAt: Instant?` в `ApexRepository` заменено на карту `smsSentMap` (время отправки хранится отдельно для каждого номера).
- `smsSentMap.clear()` вызывается при `logout()`, чтобы очищать все таймеры при выходе.
- Покрыто тест-кейсами TC-AUTH-001 (индивидуальный таймер) и TC-AUTH-002 (очистка при выходе) — оба помечены как пройденные.

# Крупные фичи

Восемь крупнейших функциональных блоков приложения (эпики из `01-analysis/5-mobile-app-spec/feature-list.md`, разделы §2.1–2.8). Для каждого — цель, найденные требования и подтверждение реализации по `implemented features.md` и коду в `android/app/src/main/java/com/apex/karting/`.

## 4. Авторизация по SMS

**Цель:** дать клиенту войти в приложение только по номеру телефона, без пароля — устранить ручную запись через Telegram/маркерную доску.

**Требования:** FR-1, UC-1, US-1 (плюс F-001…F-007 в feature-list: таймер повторной отправки, inline-ошибки, повторный вход существующего номера).

**Реализовано:** ✅ да — `RegistrationScreen.kt`. Ввод номера с маской +7, отправка и проверка SMS-кода (демо-код `1234`), обработка ошибок (неверный номер/код, превышение попыток → CTA «Связаться с поддержкой»), JWT сохраняется в DataStore для автовхода.

## 5. Расписание слотов

**Цель:** показать клиенту свободные заезды на 7 дней вперёд, чтобы он сам выбрал время без звонка администратору.

**Требования:** FR-2, FR-6, FR-11, FR-18, UC-2, UC-11, US-2, US-4, US-14 (F-008…F-017).

**Реализовано:** ✅ да — `SlotListScreen.kt`. Горизонтальный переключатель дат, карточки слотов с двумя ценами и индикатором мест, некликабельные слоты «Мест нет» / «Отменён центром», расширение диапазона дат через календарь (BS-003), состояния Loading/Error/Empty.

## 6. Детали слота

**Цель:** дать клиенту полную информацию по конкретному заезду перед бронированием, включая цену с учётом скидки.

**Требования:** FR-3, FR-6, FR-7, FR-11, UC-2, UC-3, US-3, US-7 (F-018…F-024).

**Реализовано:** ✅ да — `SlotDetailsScreen.kt`. Дата/время/конфигурация/маршал, две базовые цены, итоговая цена со скидкой лояльности, бейдж статуса лояльности, CTA «Забронировать» disabled при отсутствии мест или отмене центром.

## 7. Бронирование и оплата депозита

**Цель:** дать клиенту создать бронь и оплатить депозит по ссылке без интеграции с реальным эквайрингом — крупнейший и самый критичный сценарий (FLOW-002).

**Требования:** FR-4, FR-5, FR-8, FR-9, UC-3, UC-4, US-5, US-6 (F-025…F-034).

**Реализовано:** ✅ да — `BookingScreen.kt`. Выбор экипировки (Rental/Own) с пересчётом цены, создание брони с проверкой мест и активной брони (FR-5), обработка ошибок `409 NO_SEATS` / `409 BOOKING_ALREADY_EXISTS` / `410 SLOT_CANCELLED`, success-шторка с `depositLink`, кнопками «Оплатить депозит» и «Скопировать ссылку» (BS-002).

## 8. Жизненный цикл брони (отмена)

**Цель:** дать клиенту отменить бронь самостоятельно в разрешённое окно и увидеть причину, если центр отменил заезд сам.

**Требования:** FR-9, FR-10, FR-17, NFR-9, NFR-12, UC-5, UC-6, US-8, US-13 (F-035…F-046).

**Реализовано:** ✅ да — `BookingDetailsScreen.kt`. Статусы `Pending/Paid/Expired/Completed/CancelledByClient/CancelledByCenter` с цветовыми индикаторами, CTA «Отменить» активна при `now + 2h ≤ startsAt` (включительно), шторка подтверждения (BS-001), обработка `422 CANCEL_WINDOW_CLOSED`, success-шторка после отмены (BS-005), причина отмены центром, авто-переход в `Expired`/`Completed` в мок-репозитории.

## 9. Мои записи

**Цель:** дать клиенту видеть все свои брони в одном месте, разделённые по статусу, без обращения к администратору.

**Требования:** FR-16, UC-9, US-12 (F-047…F-049).

**Реализовано:** ✅ да — `MyBookingsScreen.kt`. Три вкладки (Активные / Состоявшиеся / Отменённые), карточки с датой/временем/статусом, empty state на каждую вкладку, обновление списка по кнопке в панели.

## 10. Профиль и лояльность

**Цель:** показать клиенту статус лояльности и скидку, дать доступ к поддержке и выходу из аккаунта.

**Требования:** FR-15, UC-8, US-11 (F-050…F-053).

**Реализовано:** ✅ да — `ProfileScreen.kt`. Телефон, бейдж статуса лояльности (None/Bronze/Silver) с размером скидки, контакты поддержки (звонок, Telegram), кнопка «Выйти». Описание условий перехода на следующий уровень лояльности не реализовано — намеренно, помечено `Won't` (нет источника правды, design-review §1.1 #4).

## 11. Оценка маршала

**Цель:** собрать обратную связь по маршалам после заезда для внутренней аналитики, без публичного рейтинга.

**Требования:** FR-12, FR-13, FR-14, FR-18, UC-7, US-10 (F-054…F-062).

**Реализовано:** ✅ да — `RateMarshalScreen.kt`. Форма из 5 звёзд без текстового поля, доступна только для завершённых заездов (`status == Completed`, `now ≥ endsAt`, `rated == false`), success-шторка «Спасибо за оценку!» (BS-006), обработка `409 ALREADY_RATED` и `422 RATING_NOT_ALLOWED`.
